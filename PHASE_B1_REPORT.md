## Classification

Per the GO prompt's options: **reader skew / state warmup / timezone mismatch / engine logic / architectural failure.**

**None of the above apply.** The actual cause is a fifth category not in the prompt:

### Harness scope misconfiguration

**Two facts make these divergences expected, not real:**

1. **None of the 4 divergent strategies are in the current burn-in portfolio.**
   ```
   23_RSI_XAUUSD_1H_MICROREV_S01_V1_P12      → 0 matches in portfolio.yaml
   03_TREND_XAUUSD_1H_IMPULSE_S01_V1_P02      → 0 matches in portfolio.yaml
   11_REV_XAUUSD_1H_SPKFADE_VOLFILT_S03_V1_P00 → 0 matches in portfolio.yaml
   17_REV_XAUUSD_1H_FAKEBREAK_S01_V1_P04      → 0 matches in portfolio.yaml
   ```
   These are strategies from PRIOR burn-in sessions. They were promoted, ran for some time, then removed from the portfolio.

2. **All 6 divergence records were written between 2026-03-25 and 2026-04-08** — weeks before TS_Engine even existed.
   ```
   First record:  written_utc 2026-03-25T23:00:00   (37 days before Phase B1 start)
   Last record:   written_utc 2026-04-08T16:26:53   (23 days before Phase B1 start)
   ```

### Mechanism

`TS_Execution/journal/SignalJournal.jsonl` is **append-only and persistent across all sessions**. It contains every signal ever generated by every strategy that has ever been in the portfolio. Today the file holds 344 records.

`TS_Engine/journal/shadow_signal_journal.jsonl` is **brand new** (created at TS_Engine sidecar startup at 05:45:42 UTC).

The parity monitor — as built — reads BOTH files in full and compares the union of (strategy_id, bar_ts) keys. Any TS_Execution record without a TS_Engine match is a "presence: PRESENT/ABSENT" divergence.

This will report as a divergence:
- **Every historical record** in TS_Execution's journal (entire backlog of weeks/months)
- **Every record from a strategy** TS_Engine doesn't load (because not in current portfolio)

Both are scope errors in the comparator, not engine drift.

## What needs to change before resume

**Comparator scope, both bounds added:**

1. **Time bound**: only compare records with `written_utc` ≥ TS_Engine's RUN_START
   ```python
   # in monitor.py, after reading TS_Execution journal:
   exec_recs = [r for r in exec_recs if r.get("written_utc", "") >= phase_b1_run_start_utc]
   ```

2. **Strategy bound**: only compare records where `strategy_id` is in TS_Engine's loaded portfolio set
   ```python
   # known set: read portfolio.yaml at monitor startup
   loaded_ids = {entry["id"] for entry in portfolio_yaml["portfolio"]["strategies"] if entry.get("enabled", True)}
   exec_recs = [r for r in exec_recs if r.get("strategy_id") in loaded_ids]
   shadow_recs = [r for r in shadow_recs if r.get("strategy_id") in loaded_ids]
   ```

Both are ~10 lines of code in `parity_monitor/monitor.py`. Neither touches engine logic, strategy logic, or live runtime.

**This is a one-time pre-soak fix to the harness, not a patch during the soak.** Per the user's rule "no code changes during soak" — the soak hadn't actually begun (the sidecar's first live bar close hadn't fired yet; first comparison was on stale historical data only).
